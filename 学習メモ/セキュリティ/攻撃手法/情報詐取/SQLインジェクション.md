# SQLインジェクション

*WebアプリケーションのWebページに存在する入力フォームやURLパラメータに、悪意のあるSQL文(データベースのデータを操作するデータベース言語)を混入させることで、データベースを不正に操作する攻撃手法*

---

## SQLインジェクション攻撃の仕組み

Webアプリケーションは、Webページ上などに設置した入力フォームなどから、ユーザーの入力値を受け取り、受け取った値を元にSQL文を動的に生成してデータベースへ問い合わせる仕組みを持っている事が多い。
その入力値に対してSQLインジェクションなどの攻撃を想定されたサニタイズ（無害化）が行われていない場合、攻撃者が意図したSQL文を「注入（inject）」する事で意図した攻撃を行う事が出来てしまう。

### 具体的な流れ(例)

1. 攻撃者がWebアプリケーションのWebページ上に設定されているログインフォームや検索フォームに悪意のある文字列を入力する
2. 入力された悪意のあるSQLがWebアプリケーションの動作に組み込まれて、入力をサニタイズする事なく、SQL文に組み込まれる
3. 本来、想定された動作によって生成されるSQLではなく、攻撃者の意図したSQL文が実行される事でデータベースを操作してしまう
4. 本来アクセスできないデータが返却・漏洩する

### 例：ログインフォームへの攻撃

本来想定されるSQL文：
```sql
SELECT * FROM users WHERE username = '入力値' AND password = '入力値';
```

攻撃者が `' OR '1'='1` という文字列を入力すると：
```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '';
```

`'1'='1'` は常に真となるため、認証を突破できてしまう。

## 情報詐取における脅威

- **個人情報の窃取**
    - 氏名・住所・電話番号・メールアドレスなど顧客DBの全件取得が可能になる
- **認証情報の漏洩**
    - パスワードハッシュや平文パスワードが格納されたテーブルを抜き出される
- **機密データへの不正アクセス**
    - 財務情報・医療情報・企業の内部文書などに到達できる
- **セッションIDの詐取**
    - セッション管理テーブルを参照し、正規ユーザーへのなりすましが可能になる

## 主な攻撃手法の分類

### 1. クラシック（インバンド）SQLインジェクション
最も一般的な手法。攻撃結果がHTTPレスポンスとしてそのまま返ってくる。

- **エラーベース**
    - データベースのエラーメッセージを意図的に発生させ、その内容からDB構造・テーブル名・カラム名を推測する
- **ユニオンベース（UNION攻撃）**
    - `UNION SELECT` 文を使い、本来の結果セットに別テーブルのデータを結合して表示させる
    - 例：`' UNION SELECT username, password FROM users--`

### 2. ブラインドSQLインジェクション
エラーメッセージや結果が直接表示されない場合でも有効な手法。

- **ブーリアンベース**
    - 真/偽の条件を送り込み、レスポンスの差異（表示の有無、レイアウトの変化）でデータを1ビットずつ推測する
- **タイムベース**
    - `SLEEP()` や `WAITFOR DELAY` などの時間遅延関数を注入し、応答時間の差異でデータを推測する
    - 例：`' AND SLEEP(5)--`（5秒遅延が発生すればSQLが実行されている証拠）

### 3. アウトオブバンドSQLインジェクション
- 攻撃結果を外部サーバー（攻撃者が用意したDNSサーバーやHTTPサーバー）に送信させる手法
- `xp_cmdshell` や `UTL_HTTP` などのDB組み込み機能を悪用する
- ファイアウォールで直接通信が制限されている環境でも情報を詐取できる

### 4. UNIONベースSQLインジェクション
- UNIONを使って、本来のクエリと別の結果を結合して表示させる攻撃です。
    ```sql
    -- 例：usersテーブルと攻撃者のクエリを結合
    SELECT name, email FROM users WHERE id = 1
    UNION SELECT username, password FROM admins;
    ```

## 対策

- **プリペアドステートメント（Prepared Statement）の使用**
    - SQL文の構造を事前に固定し、入力値をパラメータとして別途バインドすることで、SQL文の注入を根本から防ぐ
- **ORMの活用**
    - SQLを直接記述せず、ORM（Object-Relational Mapper）を通じてDBアクセスすることでリスクを低減 => ビルトインの仕様でサニタイズされている事がある
- **入力値のバリデーション・サニタイズ**
    - 入力値の型・長さ・文字種を厳格にチェックし、シングルクォートや `--`（コメント記号）などの特殊文字をエスケープする
- **エラーメッセージの非表示**
    - DBエラーをそのままユーザーに返さず、ログにのみ記録することでDB構造の露出を防ぐ
- **最小権限の原則**
    - アプリケーションがDBに接続するアカウントは、必要最低限の権限のみ付与する（参照のみ、特定テーブルのみ等）
- **WAF（Web Application Firewall）の導入**
    - SQLインジェクション特有のパターンを検知・遮断するフィルタリング層を設ける
