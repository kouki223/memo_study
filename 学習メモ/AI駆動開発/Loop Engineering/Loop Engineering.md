# Loop Engineering

参考: [もうプロンプトを書くな──「Loop Engineering」という新しいパラダイムの正体](https://zenn.dev/acrosstudioblog/articles/38509c0473683a)

## レバレッジポイントの移行

Claude Code を作った Boris Cherny 本人が「もうプロンプトは書いてない」と言い出した。

- その真意は、自身で自己解決される **Loop Engineering** に時代が向かっていることを意味する
- ただし、この loop の捉え方・扱い方が正しくないと事故が起きるため、注意が必要

## AIパラダイムの変化

| 世代 | 期間 | 名前 | 人間の仕事 | 価値の源泉 |
|---|---|---|---|---|
| 第1世代 | 2022–2024 | Prompt Engineering | 1つの指示を磨き上げる | 指示の質 |
| 第2世代 | 2025 | Context Engineering | 十分な情報・前提・例を与える | 文脈の質 |
| 第3世代 | 2026年初 | Harness Engineering | エージェントに馬具（ハーネス）を着ける | 実行環境の質 |
| 第4世代 | 2026年現在 | Loop Engineering | エージェントを自律稼働させるシステムを設計 | サイクルそのもの |

## 6つのコア

これらを loop としてフロー化し、イベント駆動的にループ処理する。**フローを決定論的に構築することが重要。**

### 1. Automations（ハートビート）

定期的、またはイベントで自動起動し、タスクを拾い上げる役。

- 毎朝の CI 失敗
- 積み上がっている issue
- 最近のコミットに混入したバグ

人間がいちいちチェックしなくてよくなり、システムが拾い上げて triage の受信トレイに書き込む。

### 2. Worktrees（防護壁）

複数のエージェントを並行で走らせるための分離レイヤー。

各エージェントが独立した git ブランチを持ち、自分の作業領域で動く。Codex と Claude Code はネイティブ対応。2つのエージェントが同じ行を同時に書き換えてコンフリクトする事故を防ぐ。

### 3. Skills（記憶チップ）

プロジェクトの仕様、ビルド手順、過去の意思決定を1つのファイルに刻んだもの。

```markdown
# .claude/skills/build.md

# Build Skill
- 依存関係インストール: `pnpm install --frozen-lockfile`
- 型チェック: `pnpm typecheck`
- テスト: `pnpm test --coverage`
- 受け入れ基準: カバレッジ80%以上、tsc エラー0件
- 過去の罠: monorepo 内で `npm` を使うと lockfile が壊れる
```

Loop が起動するたびに読み込むため、知識が貯まる。毎ラウンド最初からゼロ推論を強いられない。

### 4. Sub-agents（牽制メカニズム）

コードを書くエージェントは、ほぼ確実に自分に高得点をつける。なので「粗探し専門のエージェント」を別に走らせる。

| 役割 | 仕事 | 推奨モデル |
|---|---|---|
| Explorer | コードベース調査・前提収集 | 高速モデル |
| Implementer | 実装案を起草 | 主力モデル |
| Verifier | spec/test に照らし受け入れ判定 | **別モデル推奨** |

Verifier をあえて別モデルにすること。同じモデルだと、自分の盲点を一緒に見逃す。

### 5. Connectors（腕）

MCP（Model Context Protocol）経由で現実世界につなぐ口。Linear、GitHub、データベース、Staging API、Slack など。

Loop は「提案」だけしない。PR を作り、チケットを更新し、CI が通ったら Slack にメンションを飛ばすところまでやる。

### 6. Memory（一番過小評価されている部品）

markdown ファイル、Linear のボード、会話の外にあるあらゆる状態記録。

- モデルは忘れる。リポジトリは忘れない
- 外部状態がないと、翌日の Loop はゲームを最初からやり直す
- サボると、Loop が「毎日同じところで同じ事故を踏むやつ」に化ける
- モデルの弱点であるコンテキスト問題を解決する

## Loopに向いているタスク / 注意が必要なタスク

### 向いている

| タスク | 内容 |
|---|---|
| CIの赤信号修復 | ビルドとテストの失敗を自動で特定して直す |
| テストの補完 | 単体テストを生成・補完して通す |
| ログの巡回点検 | 異常検知、分類、修復提案 |
| ドキュメント同期 | API、README、インターフェースの自動更新 |
| 依存関係のアップグレード | バージョン確認、PR、リグレッション |

### 注意したほうがいい

| タスク | 理由 |
|---|---|
| 広範囲のリファクタリング | ロールバックが重い |
| 本番環境の権限操作 | 事故の影響が大きい |
| セキュリティに敏感な変更 | 権限ポリシー、キー、アクセス制御 |
| 純粋な美的判断 | 客観基準がなく、検証ができない |

迷ったら、まず小さく試して、証拠が出てから広げる。

## Loop 開始前に決めておくこと

| 設定項目 | 目的 |
|---|---|
| Owner（責任者） | 追跡可能性の保証 |
| 締め切り時間 | 無限ループの回避 |
| 最大ラウンド数 | コストの暴走を防ぐ |
| 証拠の出力 | ログ／スクショ／テスト／CI結果の必須化 |
| 停止理由 | デリバリー or ロールバックの明示 |

**Loop Engineering の重点は「停止条件」であって「循環回数」ではない。** 回せば偉い、ではない。

## まとめ

Boris Cherny が言っているのは、仕事が楽になったという話ではなく、**レバレッジポイントが動いた**という話。

| Before | After |
|---|---|
| 良いプロンプトを書く人 | 良いループを設計する人 |
| 個別の答えを出す力 | 答えを出し続ける機械を作る力 |
| 自分が動く | 自分の代わりに動くシステムが動く |

同じ Loop を2人のエンジニアが組んでも、出てくる結果はまったく違う。差を分けるのは、その仕事を本当に理解しているか、Loop を理解の代替にしているか。

Loop Engineering は Prompt Engineering より難しい。1つの答えを出すのと、答えを出し続ける機械を作るのは別の能力。難しい代わりに効く。
