# 宣言的プログラミング(Declarative programming)

## What is Declarative programming.

- GraphQLというDeclarative APIが存在する
    - 宣言的なAPIという意味
        - 宣言的とは、どのような意味か？

- 宣言的なプログラミング方法
    - 例：SQLやPrologなどが例として挙げられる
        - 何をやりたいか？を記述するプログラミング方法
            - どのように目的を実現するか？はコンパイラが決める

- 宣言的の対義語
    - Declarative ⇔ Imperative (命令的,指示的)
        - [Imperative vs Declarative Programming](https://fireship.dev/imperative-vs-declarative-programming)

### Deffrent Declarative to Imperative

例：コーヒーが欲しいと記述する場合

- Declarative => コーヒー頂戴
- Imperative => 細かく、コーヒーを取得するまでの流れを記述する

#### Declarative's features

- Declarativeであるという事は、内部構造や様々な事に付随する手続きを秘匿すると言う事
    - 要は、インターフェースに全てを任せるという事になる
        - インターフェースが優秀であれば使う側も優秀であり、インターフェースが優秀でなければ、優秀でなくなる

### In the real things.

- 車のオートマ(AT)のギアとマニュアル(MT)のギア
    - ギアチェンジにおいて、
        - AT車は操作が秘匿されており、状況に応じて調整されている => 宣言的
            - 運転している人はどのような処理が実際に行われているか？は把握していない
        - MT車は運転車が自分のタイミング、処理の順番で行う必要がある => 命令的
    - メリットとデメリット
        - AT
            - メリット
                - 運転者がギアのコントロールの方法を知る必要がない
                    - 手続きが秘匿されている
            - デメリット
                - 内部で不効率なギアコントロールが行われてたとしてもそれを把握する事はできない
        - MT
            - メリット
                - 運転者がギアコントロールを行う事ができる
                    - 細かなディティールを運転者の好みや状況に併せて繊細なコントロールが可能
            - デメリット
                - ギアコントロールが出来ない運転者の場合、ギアコントロールが不可能であるという事
                    - 手続きが秘匿されていない

### What is important Declarative's?

- 宣言的であるというのは、インターフェースが対象にしてる粒度という事である
    - 前項でオートマとマニュアルを比較したが、マニュアルにおいても、実際にギアがコントロールされた後の内部構造は秘匿されている
        - 対象者が車を開発するエンジニアであれば、インターフェースが変わるのである

- 宣言的であるべきか？命令的であるべきか？ => インターフェース

### Express in words Declarative

- 宣言的あるというのは、人が命令を記述しているため、完全に秘匿しているわけではない
    - 粒度が、人が理解しやすく記述しているという事(命令的である => コンピューターが理解しやすい形の命令方法)

- まとめ
```
「宣言的」なコードは「人」にフォーカスした書き方（読みやすいか）
「命令的」なコードは「コンピュータ」が理解する流れの書き方
「宣言的」かどうかのチェック観点の1つは「人が読んでサッと理解できるか」
「宣言的」に書くと可読性が上がって「開発」「保守」面で嬉しい
「宣言的」に書く上で重要な意識付けは「何をしたいか」を「適切な名詞（動詞）」で表現しようとすること
```
